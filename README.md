# 个人网盘webTable日志
## 2026-04-08 项目初始化：数据模型设计之困
**遇到的问题**：
面对“个人网盘”，不清楚需建哪些表、字段如何设计。核心纠结：用户和文件是多对多还是一对多、分享链接表是否合并到文件表、文件大小/路径等字段归属。

**尝试与纠结**：
- 最初按“每人一个文件夹”物理隔离，文件表直接存`user_id`标识归属。
- 考虑“转存”功能后，发现单文件可被多人拥有，尝试单表实现多对多导致逻辑混乱。
- 纠结分享链接表是否合并入`file`表，担心表结构过于复杂。
- 初始`file`表字段极少，随业务复杂逐步新增`path`、`file_size`、`create_time`、`ref_count`等字段；补充`originalFileName`字段，用于界面展示与文件改名。

**最终方案**：
- **用户表(`user`)**：存储账号、加密密码。
- **文件表(`file`)**：存储物理文件元数据（MD5、大小、全局存储路径）和最初上传者。
- **分享链接表(`share_link`)**：独立存放分享码、密码、有效期，通过`file_id`关联。
- **核心原则**：独立事物单表存储，外键关联，避免字段冗余。

## 2026-04-09 存储架构演进：从“每人一文件夹”到“全局MD5存储”
**遇到的问题**：
初始采用用户独立物理文件夹，转存效率低，秒传、转存功能无法高效实现，物理文件需重复复制。

**尝试与反思**：
- 物理复制方案简单但浪费空间，大文件转存速度慢，不符合商业网盘体验。
- 明确“用户文件夹”是逻辑概念，无需与物理路径绑定。

**最终方案**：
- **全局统一存储**：所有文件按MD5值存入公共目录，文件名格式为`MD5+扩展名`。
- **数据库维护逻辑归属**：`file`表`uploader_id`标识上传者，`user_saved_file`表记录所有保存者。
- **结果**：物理文件仅存一份，用户通过数据库索引共享，天然实现秒传、秒转存。

## 2026-04-10 引用计数的实现与单表适配
**遇到的问题**：
无法正确维护`ref_count`（引用计数），单表设计下主副记录计数逻辑混淆，难以实现无人使用时自动删除物理文件。

**尝试与错误**：
- 最初每条记录独立维护`ref_count`，删除时计数异常。
- 尝试用主记录`ref_count`维护存储人数，通过`ref_count>0`区分主副记录，解决初始记录查找问题。
- 纠结主记录`ref_count`初始赋值逻辑。

**最终方案**：
- **主副记录分离**：同一MD5首条记录为主记录（`ref_count>0`），秒传/转存记录为副记录（`ref_count=0`）。
- **计数维护**：仅主记录参与计数增减，删除时依据主记录`ref_count`判断是否物理删除。
- **SQL防负数**：更新语句添加`AND ref_count>0`条件，避免并发导致计数为负。

## 2026-04-11 删除逻辑的打磨与权限漏洞修补
**遇到的问题**：
- 重复调用删除接口导致`ref_count`异常递减；未校验用户权限，可越权分享、删除。
- 上传者删除主记录后，`user_id`未清空，文件仍显示且可重复删除，引发`ref_count`混乱。
- 存在越权操作、ID猜测攻击等安全风险。

**解决方案**：
- **防重放**：SQL层`AND ref_count>0`防止计数为负。
- **解除归属**：上传者删除主记录且计数>1时，`cleanower`将`user_id`置为`NULL`，隐藏文件。
- **接口权限校验**：删除、下载、分享、转存等接口，先校验用户文件归属权限。

## 2026-04-12 下载与删除的并发冲突处理
**遇到的问题**：
最后一个持有者删除文件时，若有用户正在下载，会导致下载中断、物理文件提前删除。

**尝试与讨论**：
考虑过`downloading_count`、延迟删除、文件锁等方案，不愿改动已稳定的删除逻辑。

**最终方案**：
- **复用引用计数作临时锁**：下载开始`ref_count+1`，下载结束`finally`块`-1`，确保下载期间`ref_count>1`，不触发物理删除。
- **操作系统兜底**：物理删除代码添加`try-catch`捕获`FileSystemException`，避免文件锁定导致接口异常。
- **残留文件清理**：后续通过定时任务处理，当前版本暂不实现。

## 贯穿始终的工程心法
- 先跑通核心流程，再逐步加固功能。
- 数据库设计遵循“单一职责”，通过外键关联替代字段冗余。
- 安全漏洞在测试中逐一修补，后端不信任前端传入的任何ID。
- 架构随需求演进，重构是常态，过程中解决问题是核心能力。
